From Complex to Simple

Traditional Domain (30+ lines)
# Step 1: Find categories (2 searches)
boot_catg = env['product.category'].search([
    ('name', '=', 'Boot'),
    ('level', '=', 'CatgS')
])
danner_brand = env['product.category'].search([
    ('name', '=', 'Danner'),
    ('parent_id', 'child_of', boot_catg.ids)
])

# Step 2: Find attributes (1 search)
size_values = env['product.attribute.value'].search([
    ('attribute_id.name', 'like', 'EU Shoe Size'),
    ('name', 'in', ['40', '40.5'])
])

# Step 3: Find tags (1 search)
waterproof_tags = env['product.template.tag'].search([
    ('name', 'like', 'Waterproof')
])

# Step 4: Execute query
products = env['product.product'].search([
    ('categ_id', 'child_of', danner_brand.ids),
    ('product_template_attribute_value_ids.product_attribute_value_id', 'in', size_values.ids),
    ('tag_ids', 'in', waterproof_tags.ids)
])
OQL Solution (1 line)
products = env['product.product'].searcho(
    "CatgS = 'Boot' and Brand = 'Danner' and EuShoeSize in ('40', '40.5') and Waterproof"
)

Core Concepts

1

Terms - Business Terminology

Terms map business concepts to database records. Instead of writing complex domain filters, define meaningful names that represent specific record sets.

Configuration Methods
  • Domain-Based
    Create term → Add domain rules via UI
  • Relationship Field
    Add Many2many field → Tag records through interface
Usage Examples
# Find EU Size 40 products
env['product.product'].searcho("EuShoeSize = '40'")

# Find waterproof products
env['product.product'].searcho("Waterproof")

Navigate to: Settings > Technical > OQL > Terms to configure

2

Aliases - Path Simplification

Aliases shorten long field paths into concise, memorable names. Eliminate the need to remember complex relational chains.

Common Mappings
  • product_tmpl_id.default_code → spu
  • categ_id.complete_name → category
  • partner_id.country_id.name → country
Shorthand Notation

Enable "Enable Shorthand" option for intelligent type-based resolution. OQL automatically matches value types to find correct field paths.

# With shorthand enabled for 'tags'
env['product.product'].searcho("Waterproof")
# OQL resolves path automatically!

Navigate to: Settings > Technical > OQL > Aliases to configure

3

Operator Overloading - Custom Logic

For advanced scenarios, customize operator behavior by implementing the __oql_bin__ method on your models.

class ProductAttribute(models.Model):
    _inherit = 'product.attribute'
    
    def __oql_bin__(self, term, opr, value, value_term):
        """Custom logic for term-based operations."""
        if term.domain == 'self.term_ids':
            return self.value_ids.search([
                ('id', 'in', self.value_ids.ids),
                ('name', opr, value)
            ])
        raise NotImplementedError()

Use when: Terms link to parent records but you need to query child records, or when default resolution doesn't match business requirements.

Query Syntax

Complete OQL Query Example

One query showcasing all OQL features:

FROM product.product 
SELECT name, default_code, list_price, partner_id.name
WHERE 
    CatgS = 'Boot' and 
    Brand = 'Danner' and 
    EuShoeSize in ('40', '40.5') and 
    Waterproof and 
    list_price > 1000
LIMIT 10 
OFFSET 20
  • FROM/SELECT/WHERE - SQL-like structure
  • Business Terms - CatgS, Brand, Waterproof
  • Dot Notation - partner_id.name
  • Operators - =, in, >, and
  • Pagination - LIMIT 10 OFFSET 20
  • Aliases - Auto-resolved paths
Comparison Operators
# Equality
searcho("name = 'Cold Boot'")

# Inequality
searcho("name != 'Hot Boot'")

# Greater/Less Than
searcho("age > 18")
searcho("price <= 100")

# LIKE patterns
searcho("name like 'Boot%'")

# IN clause
searcho("Size in ('40', '40.5', '41')")
Logical Operators
# AND
searcho("Brand = 'Danner' and Waterproof")

# OR
searcho("Size = '40' or Size = '40.5'")

# Combined with parentheses
searcho("(Brand = 'Danner' or Brand = 'Merrell') 
         and Waterproof")

# Complex nesting
searcho("(Size in ('40', '40.5') and Waterproof) 
         or (Size = '42' and Breathable)")
Unary Expressions

Check for existence without specifying values:

# Products that have any tags
searcho("tag_ids")

# Products with tags carrying `Waterproof` term
searcho("Waterproof")

Get Started in 3 Steps

1

Install Module

Install the OQL module and its Python dependency:

  • Install Python dependency
    pip install lark
  • Place module in addons directory
    Copy the oql folder to your Odoo addons path
  • Install via Odoo Interface
    Enable Developer Mode → Apps → Click Update Apps List → Search "oql" → Click Install
2

Configure Terms & Aliases

Set up your business terminology through the Odoo interface:

Create Terms

Navigate to:

Settings > Technical > OQL > Terms

Create terms like "Waterproof", add domain rules or link to records

Setup Aliases

Navigate to:

Settings > Technical > OQL > Aliases

Map short names to field paths, enable shorthand if needed

3

Start Querying

Use searcho() for simple WHERE clauses, or oql() for full queries:

# Simple search (WHERE only)
products = env['product.product'].searcho("Waterproof and Size = '40'")

# Full query with SELECT, LIMIT, OFFSET
results = env['product.product'].oql(
    "from product.product select name, price where active = true limit 10 offset 20"
)
You're All Set!

Best Practices

When to Use OQL
  • Complex queries involving multiple related models
  • User-facing search features where readability matters
  • Business rules that change frequently
  • Scenarios where non-developers need to understand queries
When to Use Traditional Domains
  • Simple, single-model queries
  • Performance-critical operations requiring domain optimization
  • Cases requiring very specific Odoo ORM features
Performance Considerations
  • OQL adds parsing and resolution layer
  • Term lookups are cached after first load
  • Monitor query execution times for complex alias chains
  • Use enable_shorthand judiciously
Naming Conventions
  • Use clear, business-oriented term names (e.g., EuShoeSize not eu_size_attr)
  • Keep aliases short but descriptive (e.g., spu for standard product unit)
  • Document term meanings for team reference

Advanced Features

Lazy Loading

OQL uses lazy loading for term metadata to minimize startup overhead. Terms are loaded on first use, then cached for subsequent queries.

Error Handling

Clear error messages when:

  • A term is not found
  • An alias is ambiguous
  • A field path does not exist
  • Type mismatches occur

Migration from Domains

Converting existing domain queries to OQL is straightforward:

Before (Domain)
domain = [
    '&',
    ('categ_id.name', '=', 'Boot'),
    ('tag_ids.name', '=like', 'Waterproof%')
]
After (OQL)
query = "CatgS = 'Boot' and Waterproof"
Conversion Steps
  1. Identify business concepts in your domain (these become Terms)
  2. Map field paths to shorter aliases where appropriate
  3. Replace technical field references with business terms
  4. Test thoroughly to ensure equivalent results
